Sumário Executivo

O que é

Plataforma SaaS de gestão de redes FTTH para ISPs (Internet Service Providers). Cobre desde a documentação geoespacial da rede até o monitoramento em tempo real via NOC, ordens de serviço e provisionamento de ONUs.

Mercado

+15.000 ISPs ativos no Brasil (Anatel 2024). A maioria usa planilhas Excel + WhatsApp para gestão de rede. Ticket médio de R$400–R$1.500/mês por ISP é realista. TAM estimado: R$ 72M–R$ 270M/ano só no Brasil.

Posicionamento

Competidores como OLTCloud e Geosite Telecom cobram R$800–R$3.000/mês e ainda não cobrem todo o stack. FiberOps já possui a base técnica mais completa; falta empacotar como produto e adicionar os módulos de maior valor percebido.

🔴 O que falta — Gap Analysis

Funcionalidades ausentes vs. concorrentes

Baseado na análise do OLTCloud, Geosite Telecom e necessidades reais de ISPs brasileiros.

CRÍTICO PARA RECEITA Módulos que travam a venda ▼ expandir
💳 Billing / Cobrança AUSENTE

O modelo Empresa tem status_assinatura e trial_expira_em mas não há integração com gateway de pagamento. Sem isso, a cobrança é 100% manual.

→ Integrar Stripe (internacional) ou Pagar.me / Asaas (BR) com webhook para atualizar status automaticamente.

📧 Onboarding automatizado AUSENTE

O cadastro existe, mas o fluxo pós-registro é manual (superadmin aprova). Isso impossibilita crescimento self-service. Falta: e-mail de boas-vindas, tour guiado, checklist de setup da rede.

→ Aprovação automática para trial + Resend/SendGrid para e-mails transacionais.

📱 App do Técnico (Mobile) AUSENTE

OLTCloud e Geosite destacam "Aplicativo Técnico" como diferencial. O FiberOps tem a view de campo, mas não é um PWA instalável nem app nativo. Técnicos de campo precisam trabalhar offline.

→ PWA com service worker + offline-first para ordens de serviço.

🌐 TR-069 / ACS AUSENTE

Padrão para gerenciamento remoto de CPEs (roteadores, ONTs). Permite reset de senha, atualização de firmware, diagnóstico remoto. É um produto separado no OLTCloud e cobrado à parte.

→ Integrar GenieACS (open source) como microserviço com API REST exposta no FiberOps.

ALTA PRIORIDADE Módulos que aumentam o valor percebido ▼ expandir
🔔 Notificações (WhatsApp/E-mail) AUSENTE

Alertas proativos quando ONU cai, OS criada, sinal crítico. ISPs adoram receber alerta no WhatsApp. É um dos itens mais pedidos no mercado.

→ Evolution API (WhatsApp) + Resend (e-mail) + tabela de regras de alerta por usuário.

📊 Dashboard de Interrupções AUSENTE

Detectar automaticamente quando múltiplos clientes em uma PON/CDO ficam offline simultaneamente (provável corte de fibra). Gera alerta e OS automaticamente.

→ Análise de padrões no Change Stream: X ONUs offline na mesma PON em Y minutos = interrupção.

🗃️ Backup automático AUSENTE

ISPs temem perder a documentação de toda a rede. Backup diário em S3/Cloudflare R2 do banco + exportação KMZ da rede inteira é um item de venda fácil.

→ Cron diário exportando MongoDB Atlas snapshot + exportação KMZ da rede.

🤝 Rede Neutra / Multi-provedor AUSENTE

ISPs que compartilham infraestrutura (rede neutra) precisam dar acesso restrito a parceiros. Reserva de portas, isolamento de visualização por parceiro. Cobrado como plano Enterprise.

→ Sub-projetos vinculados ao projeto-pai com permissões de leitura limitada.

📈 Relatórios exportáveis AUSENTE

PDF/Excel de: OSs do mês, ONUs offline, ocupação de CTOs, histórico de sinal. Necessário para reuniões de gestão e prestação de contas.

→ @react-pdf/renderer + exceljs com templates pré-prontos por módulo.

🔌 API pública (webhooks + REST) AUSENTE

ISPs maiores integram com ERP, billing próprio, sistemas de CRM. Uma API documentada (OpenAPI) com API keys por projeto é requisito para planos Pro/Enterprise.

→ Routes /api/v1/* com autenticação por Bearer token + documentação Swagger.

ROADMAP Módulos de diferenciação a médio prazo ▼ expandir
🤖 IA / Análise preditiva

Prever falhas por padrão de sinal degradante. "Esta ONU vai cair nas próximas 48h" com base no histórico de RX. Alto valor percebido, baixo custo de implementação com dados já coletados.

🗺️ Mapa de calor de falhas

Visualizar regiões com mais interrupções no mapa. Ajuda a priorizar expansão e manutenção preventiva da rede.

👤 Portal do cliente final

ISP dá acesso ao cliente para ver status da sua ONU, abrir chamado, ver histórico. Reduz volume de chamadas de suporte. Produto vendável como add-on.

📡 Suporte ZTE / Nokia / Fiberhome

Atualmente suporta Huawei. Ampliar para outros vendors aumenta drasticamente o mercado endereçável.

🔄 Integração com outros ERPs/CRMs

SGP já integrado. Adicionar: Voalle, Omega, iSP (sistemas de billing populares no Brasil).

📍 Roteirização de campo

Ordenar OSs do dia por rota otimizada no mapa. Reduz km rodado pelo técnico em até 40%.

🗺️ Roadmap de implementação

O que construir e em qual ordem

FASE 1 — Sprint 0–4 semanas Lançar como produto vendável

💳 Billing com Asaas

Webhook de pagamento → atualiza status_assinatura automaticamente. Planos Basic/Pro/Enterprise com trial de 14 dias free.

📧 E-mails transacionais

Resend.com: boas-vindas, confirmação, trial expirando, pagamento confirmado/falhando. Template com branding FiberOps.

🚀 Onboarding self-service

Aprovação automática para trial. Checklist de setup: "adicione sua primeira OLT → cadastre uma CTO → adicione um usuário técnico".

🔒 Limite por plano (enforcement)

O campo plano no modelo Projeto já existe. Falta enforcement: bloquear criação de OLT/usuário quando limite atingido.

FASE 2 — 1–2 meses Aumentar valor percebido

🔔 Alertas WhatsApp

Evolution API self-hosted. Quando Change Stream detecta ONU offline ou sinal crítico → mensagem para o admin/NOC do projeto.

📊 Dashboard de Interrupções

Analisar padrão no stream: 3+ ONUs da mesma PON offline em 2min = interrupção detectada. Criar OS automaticamente.

📱 PWA para técnico

Transformar /admin/campo em PWA instalável. Service worker para cache offline de OSs do dia. Push notification de novas OSs.

📈 Relatórios PDF/Excel

@react-pdf/renderer: relatório mensal de OSs, ocupação de CTOs, ONUs offline por período. Botão "Exportar" em cada listagem.

FASE 3 — 2–4 meses Módulos Premium (upsell)

🌐 TR-069 / ACS

Deploy GenieACS como microserviço Docker. API REST exposta no FiberOps para: reset de senha, reboot remoto, firmware update, relatório RSSI.

🔌 API pública v1

Routes /api/v1/* com Bearer token por projeto. OpenAPI/Swagger docs. Webhooks para: ONU offline, OS criada, provisionamento concluído.

🤝 Rede neutra

Sub-projetos parceiros com visualização restrita. Reserva de portas de OLT/CTO por parceiro. Relatório de uso por parceiro para cobrança.

☁️ Backup automático

Cron diário exportando dados do projeto para S3/R2. Histórico de 30 dias. Restauração one-click. Incluso no Pro, 90 dias no Enterprise.

FASE 4 — 4–8 meses Diferenciação e escala

🤖 IA preditiva

Modelo simples (regressão) treinado com histórico de sinal (rx_power) por ONU. Alerta "ONU deve falhar em X dias" com base em tendência.

👤 Portal do cliente final

URL pública por projeto: cliente.fiberops.com.br/[slug]. Cliente vê status da ONU, abre chamado, vê histórico. Add-on R$ 149/mês.

🏷️ White-label / Resellers

Customização de logo/cores/domínio por tenant. Programa de revendas com dashboard próprio de comissões.

📡 Multi-vendor OLT

Ampliar HuaweiOltAdapter para ZTE, Nokia, Fiberhome. Camada de abstração de adapter já existe — só adicionar implementações.

🔧 Alternativas Self-hosted

Open source equivalente ao stack do FiberOps

Para as funcionalidades ainda não implementadas, estas são as melhores opções para integrar como microserviços.

GenieACS TR-069

O ACS (Auto Configuration Server) open source mais usado no mundo. Gerencia CPEs remotamente via TR-069. API REST robusta, multi-tenant nativo.

✓Reset de senha, reboot, firmware update
✓Relatório RSSI / diagnóstico
✓API REST + webhooks

→ Deploy Docker + integrar via API no FiberOps

github.com/genieacs/genieacs

LibreNMS Monitoramento

Monitoramento SNMP completo. Templates prontos para OLTs Huawei, ZTE, Nokia. Pode complementar o NOC do FiberOps com dados SNMP.

✓SNMP + ICMP polling
✓Alertas configuráveis
✓API REST para integração

→ Complementa o NOC existente com dados SNMP

librenms.org

NetBox Inventário

IPAM + inventário de rede. Pode ser usado como "fonte da verdade" para equipamentos e sincronizado com FiberOps via API.

✓Modelagem completa de rede
✓IPAM nativo
✓Webhooks + API REST

→ Opcional — FiberOps já tem inventário próprio

netbox.dev

Evolution API WhatsApp

API REST self-hosted para envio de mensagens WhatsApp. Multi-instance, fácil integração. Principal opção para notificações no Brasil.

✓Envio de mensagens texto/mídia
✓Multi-instância (por projeto)
✓Docker + API REST simples

→ Integrar direto no noc-logger para alertas

github.com/EvolutionAPI/evolution-api

Zabbix Monitoramento avançado

Monitoramento enterprise. Templates OLT com coleta de sinal por ONU via SNMP. Mais poderoso que LibreNMS para automações complexas.

✓Templates Huawei/ZTE/Nokia prontos
✓Triggers e ações automáticas
✓Webhook para integração

→ Usar como fonte de dados SNMP para o NOC

zabbix.com

Asaas Billing BR

Gateway de pagamento brasileiro com API REST. Suporta Pix, boleto, cartão. Perfeito para cobrar ISPs no Brasil. Sandbox gratuito para testes.

✓Pix + boleto + cartão

Gargalos identificados e soluções recomendadas

Componente Problema atual Impacto Solução recomendada Prioridade
node-cron Shared event loop com Next.js. "Missed execution" a cada minuto sob carga. ALTO Migrar para worker thread separado ou BullMQ + Redis. Ou separar processo cron com node scripts/cron.js P0
SSH OLT Conexão SSH síncrona dentro do loop do cron. Bloqueia event loop por 2–10s. ALTO Pool de conexões SSH por OLT. Timeout máximo de 8s. Fila de comandos por OLT (evitar concorrência). P0
Hydration mismatch Date.now() e sparklines geradas no servidor diferem do cliente. MÉDIO Usar suppressHydrationWarning no span de hora. Sparklines só no cliente com useEffect. P1
Mongoose dup index Índice duplicado em User.email definido em index:true e schema.index(). BAIXO Remover index:true do campo ou remover o schema.index(). Manter apenas um. P2
SSE long-polling Change Stream mantém conexão MongoDB por tenant ativo. Pode exaurir o pool em escala. MÉDIO Multiplex streams por projeto usando um único watcher por collection. Broadcast para clientes do mesmo projeto_id. P1
Vercel cold start Serverless function pode ter cold start de 2–4s em acessos iniciais. MÉDIO Usar Vercel Fluid Compute (sempre ativo para rotas críticas). Ou migrar para Railway/Render para processo contínuo. P2

FiberOps AI — o diferencial competitivo de longo prazo

A maior vantagem competitiva sustentável de um SaaS de infraestrutura é a acumulação de dados históricos de rede. Com 100+ clientes, o FiberOps terá dados únicos sobre padrões de falha, comportamento de ONU, sazonalidade e muito mais — impossível de replicar por novos entrantes.

Fase 1 · Ano 1

🔔 Alertas Preditivos

Detector de anomalias em potência óptica que prevê falha de ONU 2–4 horas antes de acontecer.

  • • Coleta histórico de potência_rx por ONU
  • • Modelo estatístico: Z-score + rolling mean
  • • Alerta NOC "ONU em risco de queda"
  • • OS preventiva automática para técnico
  • • Sem GPU necessário — roda em CPU barata
  • • Tech stack: Python + scikit-learn ou simples JS

Impacto: reduz churn de clientes ISP (menos reclamação de clientes finais)

Fase 2 · Ano 2

🩺 Diagnóstico Automático

IA que analisa o NOCLog e sugere causa raiz de incidentes com base em padrões históricos.

  • • Clustering de incidentes similares
  • • "Este padrão aconteceu 3× antes — causa: cabo rompido no poste P-441"
  • • Integração com OS: abre OS automaticamente com sugestão de causa
  • • LLM local (llama.cpp) para análise de logs em texto
  • • Base de conhecimento de causas por ISP
  • • Aprende com feedback do técnico (positivo/negativo)

Impacto: MTTR (Mean Time To Repair) -40%

Fase 3 · Ano 2–3

📈 Gestão Preditiva de Rede

Modelos que prevêem crescimento de rede, saturação de PON e necessidade de expansão de infraestrutura.

  • • Previsão de saturation: "PON 4/1 estará 100% em 45 dias"
  • • Recomendação de splitter upgrade automática
  • • Análise de churn_risk por cliente (baseado em intermitência)
  • • Score de saúde de rede por região
  • • Relatório automático mensal de "Insights da sua rede"
  • • Benchmark anonimizado vs. outros ISPs do mesmo porte

Impacto: expande ticket via "AI add-on" R$99–199/mês

🔑 Autenticação da API

  • • API Key por projeto (gerada no dashboard)

  • • Rate limiting: 1.000 req/hora (Básico)
  • • Rate limiting: 10.000 req/hora (Enterprise)
  • • IP whitelist opcional por API key
  • • Logs de todas as chamadas à API
  • • Revogação instantânea de chave
  • • Múltiplas chaves por projeto

API inclusa no Pro e Enterprise. Básico: somente leitura (100 req/hora).

Glossário de termos FTTH

OLT (Optical Line Terminal)

Equipamento no data center/POP do ISP. Termina as fibras ópticas dos clientes e conecta ao backbone de internet. Marca: Huawei MA56xx, ZTE C300, Nokia 7360.

ONU/ONT (Optical Network Unit)

Modem/roteador óptico na casa do cliente. Converte sinal óptico (fibra) em elétrico (RJ45, WiFi). Identificado pelo serial único gravado fisicamente. Ex: HG8310M, EG8245H.

PON (Passive Optical Network)

Tecnologia de rede óptica passiva. Uma única fibra na OLT (porta PON) atende até 64 ONUs com splitters passivos. GPON (2.5 Gbps downstream) e XGS-PON (10 Gbps) são os padrões.

GPON (Gigabit PON)

Padrão ITU-T G.984. 2,5 Gbps downstream / 1,25 Gbps upstream compartilhados entre até 64 ONUs. Tecnologia dominante no Brasil hoje. Substituído pelo XGS-PON (10G) nas novas redes.

Splitter Óptico

Componente passivo que divide o sinal óptico de 1 fibra em N saídas. Relação 1:2, 1:4, 1:8, 1:16, 1:32, 1:64. Cada divisão adiciona ~3,5 dB de atenuação.

CTO (Caixa de Terminação Óptica)

Caixa em poste ou parede que contém splitters e fusões. Onde a fibra principal é dividida para atender os clientes do entorno. Capacidade típica: 8 a 32 portas.

CDO (Caixa de Divisão Óptica)

Ponto de derivação entre a OLT e as CTOs. Contém splitters de 1ª divisão (1:2 a 1:4). Serve uma área geográfica maior, alimentando múltiplas CTOs.

Potência Óptica (dBm)

Medida de intensidade do sinal óptico. ONU Rx saudável: -8 a -27 dBm. Abaixo de -27 dBm: sinal fraco, pode haver queda. Abaixo de -30 dBm: perda de link iminente.

Provisionamento

Processo de configurar uma ONU na OLT para que ela passe tráfego. Envolve adicionar o serial da ONU, definir T-CONT (largura de banda), GEM port (serviço), e VLAN. Sem provisionamento, a ONU fica "offline" mesmo fisicamente conectada.

T-CONT (Transmission Container)

Alocação de banda upstream para uma ONU na OLT. Define burst size, fixed bandwidth, assured bandwidth. Um T-CONT por serviço (PPPoE, VoIP, IPTV).

TR-069 / CWMP

Protocolo de gerenciamento remoto de CPEs (roteadores). O roteador do cliente conecta a um servidor ACS (Auto Configuration Server) e pode ser configurado, monitorado e atualizado remotamente. Padrão Broadband Forum.

SNMP (Simple Network Management Protocol)

Protocolo padrão para monitoramento de equipamentos de rede. A OLT expõe MIBs (Management Information Base) com dados de status. Usado por Zabbix, LibreNMS, Grafana para monitoramento.

🔴 Fix #1 — node-cron "missed execution" warning

P0 · 2–4 horas

Problema: node-cron compartilha o mesmo event loop do Next.js. Quando o servidor está ocupado (request handling, compilação), o cron perde execuções.

Solução A — Worker Thread (mais simples)

// scripts/cron-worker.js (novo arquivo)
const { Worker, isMainThread, parentPort } = require('worker_threads')

if (isMainThread) {
  // Exportar para usar no instrumentation.js
  module.exports = function startCronWorker() {
    new Worker(__filename)
  }
} else {
  // Executa em thread separada — sem bloquear event loop
  const cron = require('node-cron')
  require('@/lib/db').connectDB()

  cron.schedule('*/5 * * * *', async () => {
    const { syncSGP } = require('@/lib/sgp-sync')
    // ... jobs aqui
  })
}

// instrumentation.js — trocar para:
import { startCronWorker } from './scripts/cron-worker'
startCronWorker() // roda em thread separada

// src/lib/olt/adapters/huawei.js
import { NodeSSH } from 'node-ssh'
export class HuaweiAdapter extends OLTAdapter {
  async connect() {
    this.ssh = new NodeSSH()
    await this.ssh.connect({ host: this.host, username: this.config.user,
      password: this.config.password, readyTimeout: 10000 })
    // Entrar no modo enable + config
    await this.ssh.execCommand('enable')
  }
  async provisionONU({ frame, slot, port, onuId, serial, profile, vlanId, description }) {
    const cmds = [
      `interface gpon ${frame}/${slot}`,
      `onu add ${port} ${onuId} sn-auth ${serial} omci ont-lineprofile-id ${profile}`,
      `port vlan eth ${port} ${onuId} 1 vlan ${vlanId}`,
      `onu desc ${port} ${onuId} desc ${description}`
    ]
    for (const cmd of cmds) {
      const r = await this.ssh.execCommand(cmd)
      if (r.stderr) throw new Error(\`Huawei CLI error: \${r.stderr}\`)
    }
    return { success: true }
  }
}

// src/lib/olt/factory.js — Padrão Factory para selecionar adapter
export function createOLTAdapter(olt) {
  const adapters = { huawei: HuaweiAdapter, zte: ZTEAdapter, datacom: DatacomAdapter }
  const Adapter = adapters[olt.vendor?.toLowerCase()]
  if (!Adapter) throw new Error(\`Vendor não suportado: \${olt.vendor}\`)
  return new Adapter(olt)
}

// src/lib/olt/connection-pool.js
import { LRUCache } from 'lru-cache'

const pool = new LRUCache({
  max: 50,                    // máx 50 OLTs conectadas simultaneamente
  ttl: 5 * 60 * 1000,         // desconecta após 5 min de inatividade
  dispose: async (adapter) => { await adapter.disconnect() }
})

export async function getConnection(oltId, config) {
  if (pool.has(oltId)) return pool.get(oltId)
  const adapter = createOLTAdapter(config)
  await adapter.connect()
  pool.set(oltId, adapter)
  return adapter
}

// Webhook Asaas recebe evento de inadimplência
// POST /api/webhooks/asaas
export async function POST(req) {
  const { event, payment } = await req.json()
  const hmac = req.headers.get('asaas-signature')
  if (!verifyHMAC(hmac, payload, process.env.ASAAS_WEBHOOK_SECRET)) {
    return Response.json({ error: 'Invalid signature' }, { status: 401 })
  }

  if (event === 'PAYMENT_OVERDUE') {
    const cliente = await Cliente.findOne({ asaas_customer_id: payment.customer })
    if (!cliente) return Response.json({ ok: true })

    const diasAtraso = differenceInDays(new Date(), new Date(payment.dueDate))

    if (diasAtraso >= 15 && cliente.projeto.auto_suspend_enabled) {
      // Suspender ONU via SSH
      const onu = await ONU.findOne({ cliente_id: cliente._id })
      if (onu) {
        const adapter = await getConnection(onu.olt_id)
        await adapter.suspendONU(onu.serial)
        await NOCLog.create({
          projeto_id: cliente.projeto_id,
          tipo: 'SUSPENSION',
          mensagem: \`ONU \${onu.serial} suspensa por inadimplência (\${diasAtraso} dias)\`,
          onu_serial: onu.serial
        })
      }
    }
  }

  if (event === 'PAYMENT_RECEIVED') {
    // Reativar automaticamente
    const cliente = await Cliente.findOne({ asaas_customer_id: payment.customer })
    if (cliente?.status === 'suspended') {
      const onu = await ONU.findOne({ cliente_id: cliente._id })
      if (onu) {
        const adapter = await getConnection(onu.olt_id)
        await adapter.reactivateONU(onu.serial)
        await Cliente.updateOne({ _id: cliente._id }, { status: 'active' })
      }
    }
  }
}

// FASE 2 — Criptografia em repouso com chave por tenant:
import { createCipheriv, createDecipheriv, randomBytes } from 'crypto'
const ENCRYPTION_KEY = Buffer.from(process.env.ENCRYPTION_KEY_HEX, 'hex') // 32 bytes

export function encryptSecret(plaintext) {
  const iv = randomBytes(16)
  const cipher = createCipheriv('aes-256-gcm', ENCRYPTION_KEY, iv)
  const encrypted = Buffer.concat([cipher.update(plaintext, 'utf8'), cipher.final()])
  const authTag = cipher.getAuthTag()
  return iv.toString('hex') + ':' + encrypted.toString('hex') + ':' + authTag.toString('hex')
}

export function decryptSecret(ciphertext) {
  const [ivHex, encryptedHex, authTagHex] = ciphertext.split(':')
  const decipher = createDecipheriv('aes-256-gcm', ENCRYPTION_KEY, Buffer.from(ivHex, 'hex'))
  decipher.setAuthTag(Buffer.from(authTagHex, 'hex'))
  return decipher.update(Buffer.from(encryptedHex, 'hex')) + decipher.final('utf8')
}
